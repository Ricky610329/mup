# MUP 設計哲學

本文件解釋 MUP 設計背後的原則 — 為什麼做了某些選擇，以及更重要的，為什麼刻意不做某些功能。[規格](./MUP-Spec.zh-TW.md)定義了「是什麼」和「怎麼做」；本文件涵蓋「為什麼」。

---

## 三項原則

### 1. 安全性是 host 的責任

協議不定義任何安全機制。MUP 宣告它需要哪些權限；host 決定是否授予。iframe 沙箱、權限策略、內容隔離 — 這些全部是 host 的事。

這讓協議保持簡潔，並讓每個 host 按照自身環境實現適當的安全措施。本機開發用的 host 和正式部署的 host 會有截然不同的安全需求；協議不應該規定任何一方。

### 2. MUP 提供函式與函式描述

MUP 是自描述的。它的 manifest 宣告了它做什麼（`description`）和它能做什麼（帶有描述與 JSON Schema 的 `functions`）。LLM 看不到 UI、CSS 或 HTML — 它只看到這些描述。

好的描述是視覺介面與語言模型之間的橋樑。協議確保這座橋存在（描述是必填欄位）；但無法確保這座橋建得好（那是 MUP 作者的工作）。

### 3. 渲染不受限制

MUP 是一份完整的 HTML 文件，在 iframe 中渲染。它可以使用任何 Web 技術 — Canvas、SVG、WebGL、影片、音訊或純 DOM。協議不規定內容如何渲染。

一份小小的 HTML 檔案可以引用外部函式庫來構建複雜的介面。協議不禁止這樣做 — 但 host 不保證提供網路存取，因此自包含能提升可攜性。

## 核心設計決策

### 單一檔案，無需建置

一個 MUP 就是一個 `.html` 檔案。拖進 host 就能運作。manifest、UI 和邏輯全部在同一個檔案裡。這是建立和分享 MUP 最低的門檻。

### 共用函式

在 manifest 中宣告的函式可以被 LLM 呼叫（作為工具）或被使用者觸發（透過 UI）。不是每個函式都需要對應一個按鈕，也不是每個 UI 操作都需要對應一個函式 — 但當兩者重疊時，實作是共用的。這就是 MUP 雙向互動的基礎：協議透過同一個機制支持兩種呼叫者。

### LLM 作為協調者

MUP 之間不互相通訊。LLM 讀取一個 MUP 的輸出，決定是否呼叫另一個。每個 MUP 只需要知道自身和 host。跨 MUP 的協調是 LLM 的工作。

### Host 無關性

規格定義了 MUP 與 host 之間的訊息契約。它不定義 host 如何渲染 MUP、如何向 LLM 呈現資訊、或如何實施安全措施。不同的 host 可以使用不同的佈局、不同的 LLM API、不同的隔離策略。協議在所有情境下都能運作。

### SDK 由 host 注入

`mup` 全域物件不是 MUP 引入的函式庫 — 它是 host 在載入時注入的。MUP 零依賴，不需要追蹤 SDK 版本。host 控制 SDK 的實作，確保協議相容性。

## 刻意不做的功能

以下每項功能都經過考量並刻意排除。論證遵循一個原則：**每一個協議功能都是對所有實作者徵收的稅。** 一個只幫助 10% MUP 但增加 100% host 複雜度的功能，不值得納入協議。

| 功能 | 為何不做 | 替代方案 |
|------|----------|----------|
| **動態 manifest** — 執行時增減函式 | 強制每個 host 管理對話中途的工具列表更新和 LLM 重新同步。 | 提供一個通用函式（如 `execute(command, args)`），在描述中說明可用的命令。 |
| **函式取消** — 中止執行中的呼叫 | 強制每個 MUP 作者處理中止邏輯、部分狀態和競態條件。 | 函式執行到完成為止。無法在保持 MUP 存活的情況下取消單次呼叫 — 如果必須完全停止 MUP，host 透過 shutdown 銷毀容器。 |
| **Host 能力查詢** — 「你支不支持 X？」 | 需要一套共用的功能名稱詞彙表，限制 host 的擴展方式。 | 嘗試操作；處理 `-32601`（Method not found）錯誤。 |
| **輸出 schema** — 宣告函式回傳格式 | 對每個函式增加撰寫負擔，且效益不明確。回傳格式已是結構化的（帶型別的 content items）；為每個函式再加 output schema，收益不大。 | 在函式的 `description` 中描述預期輸出。 |
| **進度回報** — 呼叫期間的中間更新 | 強制 host 決定如何以及向何處轉發進度。 | 使用 `updateState()` 在長時間操作期間更新視覺 UI。函式呼叫保持簡單：請求進，回應出。此外，MUP 可以在 manifest 中宣告 `notifications` 欄位，請求即時通道推送、輪詢或靜默模式 — 給 host 一個提示，說明狀態變更應以多高的緊迫程度傳達給 LLM。 |
| **二進制傳輸** — 避免 base64 開銷 | 使訊息層變得複雜。JSON-RPC 2.0 簡單、可除錯且通用。 | 使用 `data` content type 攜帶任何你需要的編碼。base64 適用於圖片；自訂格式可以作為結構化 JSON 嵌入。 |
| **串流** — 呼叫中的漸進式資料傳遞 | 增加一種新的通訊模式，帶來自身的流量控制和錯誤處理。 | 函式的參數和結果通常很小。大量資料應由 MUP 自行取得（如透過 HTTP）。MUP 對 host 的串流可透過 `updateState()` 和 `notifyInteraction()` 實現 — 快速回傳函式呼叫，再透過 notification 持續推送更新。Host 對 LLM 的串流完全是 host 自己的事。 |
| **MUP 間通訊** — 直接傳訊 | 在 MUP 之間建立依賴圖，破壞自包含性。 | LLM 居中協調。host 可以將跨 MUP 通訊作為 host 專屬擴展提供。 |

## 評估未來變更

考慮新的協議功能時，套用以下測試：

**1. 協議稅。** 這會不會對所有 host、所有 MUP 或兩者都增加工作 — 即使那些不使用此功能的？如果會，門檻要非常高。

**2. 自由度測試。** 加入這個功能會不會限制 host 或 MUP 的實作方式？能力註冊表限制 host 只能使用固定的詞彙。取消機制限制 MUP 作者必須撰寫中止邏輯。限制會累積。

**3. 替代方案測試。** 同樣的目標能不能用現有機制達成？如果合理的替代方案存在，協議就不需要這個功能。

**4. 列舉值 vs. 機制。** 在現有列舉中加一個新值（如新的 content type）成本低。加一種新的通訊模式或生命週期階段成本高。優先選前者。

通過全部四項測試的變更 — 低稅、不限制自由、無法用替代方案解決、且小到只是一個列舉值 — 是好的候選項。其他一切應留給 host 專屬擴展。
